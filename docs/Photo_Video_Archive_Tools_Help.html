Overall Goal of These Tools

The purpose of this toolset is to take a messy photo/video archive and turn it into a structured, safer-to-review, de-duplicated, metadata-aware archive.

The tools are designed for a common real-world situation:

  • You have some files already partially sorted into folders or albums.
  • You have many loose files in a general holding folder such as Needs Categorized.
  • Some files have good embedded dates, some have missing/bad metadata, and some are older videos where embedded metadata cannot be safely written.
  • Many files may share original camera names such as IMG_0001.JPG or MOV_0001.MOV, so dumping everything into one folder would risk overwrites and make manual comparison harder.
  • You need unique filenames before you can safely consolidate, compare, de-dupe, tag, and organize.
Core strategy: rename first to create unique stable filenames, use folder context carefully for files missing dates, remove confirmed duplicates, then run the corrective metadata/tag/sidecar pass after the archive structure is stable.

What problem this solves

A large personal media archive usually has several overlapping problems: repeated camera filenames, partial manual organization, exported or downloaded copies, recompressed videos, missing EXIF dates, old video formats, and broad catch-all folders. These tools separate that problem into controlled stages.

Why rename first?

Renaming gives every supported media file a unique archive-safe name:

YYYY-MM-DD_0000000001_IMG.ext
YYYY-MM-DD_0000000002_VID.ext

This makes later steps safer because files no longer depend on fragile camera names that may repeat across devices, exports, downloads, or folders.

Why not immediately dump everything into one folder?

For large archives, dumping everything into one folder can create overwrite risk and an overwhelming manual review problem. A safer approach is to keep partially sorted folders, move unknown items into a general holding folder, then normalize filenames across the whole archive.

Why manually group some files before CatchUp?

CatchUp can infer dates from already-renamed files in the same immediate folder, but only when that folder provides meaningful context. For example, an undated file placed into a folder full of already-renamed photos from one trip or event has much better date context than an undated file sitting in a giant mixed-year holding folder.

Important: CatchUp is not intended to magically date unrelated files in a broad mixed folder. With the default safeguards, broad mixed-date folders should be rejected if the date span is too wide. If you force or loosen the safeguards, inferred dates may reflect the broad folder context rather than the true event date.

Why de-dupe after renaming and grouping?

Duplicate detection is easier and safer after filenames are unique and the archive structure is closer to its final form. Image and video duplicate review can then compare files across the chosen scan root, while the reviewed CSV provides a controlled delete process where only confirmed delete rows are processed.

Why run the corrective script last?

The corrective script uses the current parent folder name as the tag/keyword source. If you run it before the archive is organized, tags may reflect temporary folders instead of meaningful final folders. Running it last lets the script set image tags and video sidecars based on the final folder structure.

End result

After the full workflow, the archive should have:

  • Unique archive filenames.
  • Reduced duplicates.
  • More reliable filename-derived dates for files that had missing metadata.
  • Parent-folder tags on images where supported.
  • XMP sidecars for videos, including older formats where embedded metadata writing is not practical.
  • A clearer folder structure that can be imported into other tools more cleanly.

Why Use a Needs Categorized Holding Folder?

A Needs Categorized folder is useful when you already have some meaningful album/event folders, but you also have a large set of loose or unknown photos and videos that are not ready to place yet.

Best use: treat Needs Categorized as a safe staging area. It lets you keep unknown files separate, rename them into unique archive-safe filenames, and review them without mixing them into existing album folders too early.

Why this helps when you already have existing album folders

If your archive already contains folders such as trips, holidays, family events, school events, or old camera imports, those folders may contain useful context. Keeping those existing folders intact has several benefits:

  • It avoids dumping thousands of files into one overwhelming folder.
  • It reduces overwrite risk from repeated camera filenames such as IMG_0001.JPG or MOV_0001.MOV.
  • It preserves clues from the current folder structure while you normalize filenames.
  • It lets already-sorted folders become useful context for later CatchUp date inference.
  • It gives you a controlled place to put files that still need human review.

Recommended pattern

  1. Keep existing meaningful folders where they are.
  2. Move unknown or loose files into Needs Categorized.
  3. Run the rename script so files receive unique archive-safe names where metadata dates are available.
  4. Review files that still could not be dated or renamed.
  5. Move identifiable undated files into the correct related album/event folder.
  6. Run CatchUp after that manual grouping so same-folder date inference has meaningful context.
  7. Run duplicate detection and corrective metadata/tagging after the archive structure is closer to final.

How this interacts with CatchUp

CatchUp can estimate a date from already-renamed files in the same immediate folder. That is powerful only when the folder is meaningful. For example:

D:\MediaArchive\Photos and Videos\Beach Trip - 2020
  2020-07-14_0000001234_IMG.jpg
  2020-07-14_0000001235_IMG.jpg
  2020-07-15_0000001236_VID.mp4
  IMG_0042.JPG

In that example, IMG_0042.JPG may be a reasonable CatchUp candidate because the folder contains related already-renamed files with a narrow date range.

By contrast:

D:\MediaArchive\Photos and Videos\Needs Categorized
  2012-05-01_0000000100_IMG.jpg
  2018-11-20_0000002222_IMG.jpg
  2024-03-10_0000009876_VID.mp4
  IMG_0042.JPG

That folder is too broad to be a reliable date source. The correct move is to manually identify IMG_0042.JPG, move it into the applicable related folder, or manually confirm its date through the CSV.

How this interacts with the corrective script

The corrective script adds the immediate parent folder name as an image tag or video sidecar tag. Because Needs Categorized is a temporary staging folder, the script intentionally avoids adding that folder name as a tag. After files are moved into meaningful folders, rerun the corrective script so tags and sidecars reflect the final folder names.

Quick Answers From the Latest Changes

Does ffmpeg_paths.ps1 have to exist?

No. The video finder auto-detects the local FFmpeg install from .\ffmpeg\bin\ffmpeg.exe and .\ffmpeg\bin\ffprobe.exe. The ffmpeg_paths.ps1 file is only an optional helper created by the installer after it verifies the paths.

You can delete ffmpeg_paths.ps1 and the current Python tools should still work. If you rerun the FFmpeg installer, it may recreate the file.

Does -ResetExistingTags also update video sidecars?

Yes, in the current ResetTags-capable corrective script, -ResetExistingTags clears/reset image tag fields and also rewrites existing video sidecars so stale sidecar tags are replaced with the current folder tag/date values.

You do not need to also pass -UpdateExistingSidecars when running a reset pass. Use -UpdateExistingSidecars by itself when you only want to rewrite existing video sidecars without doing the broader image tag reset.

Verify your corrective script has the reset-capable options

python .\Correct-ImageVideoMetadataFromFilename.py -h | findstr /I "ResetExistingTags UpdateExistingSidecars"

If ResetExistingTags does not appear, replace the corrective script with the current ResetTags-capable version before relying on reset behavior.

Step-by-Step Example Workflow: Creating and Starting From Scratch

This workflow is for the first time you are cleaning up a large messy archive. It is written as a teaching workflow, not just a list of commands. The exact order matters because each step makes the next step safer.

Example scenario: you have a large archive with some files already partially sorted into folders, plus a large general holding folder such as Needs Categorized. Some files have reliable dates, some do not, and many old camera filenames may repeat.

Step 0 — Install prerequisites and open PowerShell in the tool folder

Before touching the archive, make sure Python, required Python packages, ExifTool, and FFmpeg are available. Then run all commands from the tool folder so logs, state files, reports, and caches stay together.

Set-Location "C:\MediaArchiveTools"

python --version
python -m pip show Pillow
python -m pip show send2trash
& "C:\Tools\ExifTool\exiftool.exe" -ver
.\ffmpeg\bin\ffmpeg.exe -version
.\ffmpeg\bin\ffprobe.exe -version

Why this matters: the rename and corrective tools need ExifTool; the image/video reviewers need Python image packages; the video duplicate finder needs FFmpeg/FFprobe.

Step 1 — Start with your existing partial organization

Do not immediately dump everything into one massive folder. Keep folders that are already meaningful, such as event folders, trip folders, family folders, or partially sorted albums. Put truly unknown files into a general holding folder such as Needs Categorized. See Why Use a Needs Categorized Holding Folder? for the reason this staging folder is useful.

D:\MediaArchive\Photos and Videos
  Birthday Party - 2018
  Beach Trip - 2020
  Old Phone Export
  Needs Categorized

Why this matters: many cameras and phones reuse names like IMG_0001.JPG. Keeping the starting folders avoids overwrite risk and preserves clues that may help with later manual review.

Step 2 — Preview the initial rename

The first rename pass reads metadata dates where available and gives files unique archive-safe filenames. Run -WhatIf first so you can see what would happen without changing anything.

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -WhatIf

Why this matters: once files have unique names, they are much safer to move, compare, review, and de-dupe. This reduces the risk of overwriting repeated camera filenames.

Step 3 — Run the initial rename live

If the preview looks correct, run the rename for real.

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos"

Result: files with usable metadata dates are renamed into the archive format. Files without usable dates remain unresolved and may be written to a review CSV depending on the options used.

Step 4 — Review files that still did not get renamed

After the initial rename, focus on the remaining non-renamed files. These are usually the files with missing/bad metadata or unsupported date fields.

At this point you have three choices:

SituationRecommended actionWhy
You can tell what event/folder the file belongs to.Manually move it into the applicable folder with related already-renamed files.This gives CatchUp meaningful same-folder date context.
You cannot identify the file yet.Leave it in Needs Categorized or manually enter a date in the review CSV.Guessing from a broad mixed folder is risky.
The file belongs to a new event not already represented.Create a specific folder for that event/date range if known, or use manual CSV date confirmation.Specific folders produce safer tags and safer date inference later.
Do not treat a giant holding folder as reliable date context. If a folder contains years of unrelated files, any inferred date from that folder is much less meaningful. Use manual grouping or manual CSV confirmation instead.

Step 5 — Run CatchUp after manual grouping

CatchUp is for the remaining unrenamed files after you have done some manual grouping. It first asks ExifTool for dates on only the leftover unrenamed files. If ExifTool still cannot find a date, CatchUp can infer a date from already-renamed files in the same immediate folder, subject to safeguards.

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -CatchUp `
  -CatchUpMinFolderDates 3 `
  -CatchUpMaxFolderDateSpanDays 45 `
  -CatchUpReviewCsv ".\photo_rename_catchup_missing_dates.csv" `
  -WriteMissingCsvInWhatIf `
  -WhatIf

If the preview is correct:

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -CatchUp `
  -CatchUpMinFolderDates 3 `
  -CatchUpMaxFolderDateSpanDays 45 `
  -CatchUpReviewCsv ".\photo_rename_catchup_missing_dates.csv"

Why you would do CatchUp: it avoids rescanning the whole completed archive with ExifTool and focuses only on leftover unrenamed files. It also uses the folder context you created manually.

When not to rely on CatchUp: if the file is still sitting in a broad mixed folder and the folder contains unrelated dates, use the review CSV and manually set the date instead.

Step 6 — Process any CatchUp review CSV rows manually

If CatchUp still cannot safely date some files, it writes them to a CSV. Open the CSV, review the file, and set:

Decision = CONFIRM
DateToUse = YYYY-MM-DD

Then preview and process that CSV:

python .\Rename-PhotosVideos-ExifTool.py `
  -ReviewCsv ".\photo_rename_catchup_missing_dates.csv" `
  -WhatIf

python .\Rename-PhotosVideos-ExifTool.py `
  -ReviewCsv ".\photo_rename_catchup_missing_dates.csv"

Why this matters: manual date confirmation is safer than letting a broad folder guess dates for unrelated files.

Step 7 — Decide whether to organize more before de-dupe

At this point, the filenames are unique and most dates should be represented in the filenames. If your files are still mostly in a general holding folder, this is a good time to do more manual organization into meaningful subfolders.

If your files are already in meaningful folders: continue to duplicate detection.

If your files are not organized yet: use the new unique filenames and date information to create folders first. After folder organization, run the corrective script later so tags reflect the final parent folder names.

Step 8 — Find image duplicates

Run the image duplicate finder after filenames are stable and after you have consolidated enough folders under one scan root for duplicates to be compared.

python .\Find-SimilarImages-ReviewDelete.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -OutputFolder ".\duplicate_reports" `
  -Workers 16

Why this matters: the image finder can detect visually similar copies even when files have been renamed, resized, or recompressed.

Step 9 — Find video duplicates

Run the video duplicate finder separately. Video comparison is more expensive because it samples frames with FFmpeg.

python .\Find-SimilarVideos-ReviewDelete.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -OutputFolder ".\duplicate_reports" `
  -SamplePoints "0.25,0.50,0.75" `
  -FFmpegTimeoutSeconds 8

Why videos are separate: videos may have different compression, resolution, or file size while still showing the same content. The video finder uses duration and sampled frames rather than relying only on filename or file size.

Step 10 — Review duplicate CSVs in the central reviewer

Open the image or video review CSV in Review-SimilarFiles.py. The reviewer does not delete anything. It only helps you mark keep/delete decisions and save a reviewed CSV.

python .\Review-SimilarFiles.py `
  -Csv ".\duplicate_reports\<timestamp>_similar_images_review.csv"

python .\Review-SimilarFiles.py `
  -Csv ".\duplicate_reports\<timestamp>_similar_videos_review.csv"

Why this matters: the review step is the safety gate. The delete processors only act on rows where SuggestedAction=DELETE and ConfirmDelete=CONFIRM.

Step 11 — Process confirmed duplicate deletes with WhatIf first

After saving a reviewed CSV, run the processor in -WhatIf mode first.

python .\Find-SimilarImages-ReviewDelete.py `
  -Process `
  -Csv ".\duplicate_reports\<reviewed_image_csv>.csv" `
  -WhatIf

python .\Find-SimilarVideos-ReviewDelete.py `
  -Process `
  -Csv ".\duplicate_reports\<reviewed_video_csv>.csv" `
  -WhatIf

Why this matters: -WhatIf shows what would be deleted before anything is sent to the Recycle Bin.

Step 12 — Process confirmed duplicate deletes live

If the -WhatIf output is correct, run the live processor.

python .\Find-SimilarImages-ReviewDelete.py `
  -Process `
  -Csv ".\duplicate_reports\<reviewed_image_csv>.csv"

python .\Find-SimilarVideos-ReviewDelete.py `
  -Process `
  -Csv ".\duplicate_reports\<reviewed_video_csv>.csv"

Result: confirmed duplicate files are sent to the Recycle Bin by default. For videos, matching video_file.ext.xmp sidecars are also processed by default.

Step 13 — Run the corrective metadata/tag/sidecar pass last

Run the corrective script after the archive is organized and de-duplicated. This is the final pass that uses filenames and parent folders to repair metadata/tag structure.

python .\Correct-ImageVideoMetadataFromFilename.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -ResetExistingTags `
  -WhatIf

python .\Correct-ImageVideoMetadataFromFilename.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -ResetExistingTags

Why this is last: the corrective script uses the current parent folder as the tag source. Running it after organization means your tags reflect the final folder structure, not temporary import folders or holding folders.

Alternative path — if nothing is organized yet

If your archive starts as one mostly unorganized folder, use the rename script first to create unique filenames. Then use filename dates, previews, and manual review to organize files into meaningful subfolders. After that, run duplicate detection and the final corrective script.

Important for undated files: if undated files are not manually grouped into a meaningful folder, folder-based inference may be weak or rejected by the safety rules. Do not rely on a giant mixed folder to provide accurate dates.

Finished state

When complete, you should have a structured archive with stable filenames, reduced duplicates, corrected image dates/tags where supported, and video sidecars for date/tag metadata.

Adding New Files/Folders Later

After the archive is already cleaned up, you usually do not need to rerun every step across the entire archive for every new batch. The best approach depends on what you are adding.

General rule: process the smallest folder that contains the new files, unless you intentionally need duplicate comparison against the entire archive.

Scenario A — Adding a new import folder

Use this when you copied new photos/videos into a temporary import folder such as:

D:\MediaArchive\Photos and Videos\New Import - Phone 2026

Recommended steps

  1. Run the rename script only against the new import folder.
  2. Review any missing-date CSV rows.
  3. Move the newly renamed files into final album/event folders.
  4. Run the corrective script against the folders you changed.
  5. Run duplicate detection against the full archive only if you suspect these files duplicate older files elsewhere.
python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos\New Import - Phone 2026" `
  -WhatIf

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos\New Import - Phone 2026"

Why not scan the whole archive? the rename script skips already-renamed files, but targeting the new folder is still faster, cleaner, and easier to audit.

Scenario B — Adding files directly into an existing event folder

If you know the files belong in an existing folder, place them there first. Then run rename/CatchUp against that folder only.

python .\Rename-PhotosVideos-ExifTool.py `
  -Root "D:\MediaArchive\Photos and Videos\Beach Trip - 2020" `
  -CatchUp `
  -CatchUpMinFolderDates 3 `
  -CatchUpMaxFolderDateSpanDays 45 `
  -WhatIf

Why this works well: the existing folder already contains related renamed files, so CatchUp has meaningful same-folder date context for any new files missing metadata.

Scenario C — Adding a broad unsorted batch

If you add hundreds or thousands of unknown files, treat that batch like a smaller version of the original cleanup:

  1. Place them in a new holding folder such as Needs Categorized - Import 2026-06.
  2. Run rename against that holding folder.
  3. Manually move unresolved files into meaningful folders where possible.
  4. Run CatchUp only after that manual grouping.
  5. Run duplicate detection against either the import folder or the full archive depending on your goal.
  6. Run the corrective script after final placement.
Do not use CatchUp as a broad-date guesser for a mixed holding folder. If the import folder contains unrelated years/events, manually group files first or confirm dates through the CSV.

Scenario D — Checking duplicates only inside the new batch

If you only care whether the new batch has duplicates within itself, run duplicate detection against the new import folder only.

Scenario E — Checking duplicates against the full archive

If the new files may duplicate older files already in the archive, run duplicate detection against the full archive root.

python .\Find-SimilarImages-ReviewDelete.py `
  -Root "D:\MediaArchive\Photos and Videos" `
  -OutputFolder ".\duplicate_reports" `
  -Workers 16

python .\Find-SimilarVideos-ReviewDelete.py `

Why choose this: it is slower, but it can find duplicates between the new files and older archive files. The caches help future runs avoid reprocessing unchanged media.

Scenario F — After moving or renaming folders

If you reorganize folders after files were already tagged or sidecars were already written, rerun the corrective script against the changed folder or the full archive so parent-folder tags and sidecars match the new structure.

  -WhatIf

Use a specific folder when: you only changed one event/album folder.

Use the full archive root when: you reorganized many folders or want to normalize everything again.

Review-SimilarFiles.py

This is the browser-based central review tool for both image and video duplicate groups. It does not delete, move, or rename files. It only updates the review CSV state and saves a reviewed CSV.

Current Features

  • Opens image review CSVs and video review CSVs.
  • Multiple KEEP rows per group are allowed.
  • Text filter searches every filename/path in the group.
  • Video preview uses the browser where the video container/codec is supported.
  • Open file and Open folder buttons are available for files the browser cannot preview.
  • Suppresses harmless browser-aborted video stream traceback noise.
  • Confirm All Remaining Groups confirms all DELETE rows currently shown by filters.

PowerShell path quoting

python .\Review-SimilarFiles.py `
  -Csv "C:\MediaArchiveTools\duplicate_reports\<review_csv>.csv"
PowerShell treats & as a special character. Any path containing MediaArchiveTools must be wrapped in quotes.

Reviewer-side video keeper preference

Current video keeper preference:
  1. If all files in the group are in the same directory and have the same non-zero file size, keep the file with the older Date Modified.
  2. Otherwise prefer an archive-renamed video filename.
  3. Then prefer a file with a sidecar.
  4. Then prefer higher estimated bitrate / larger effective file size.
  5. Then prefer higher resolution.
  6. Then shorter path / alphabetical path.

How KEEP / DELETE Suggestions Are Determined

The duplicate finders and reviewer make suggestions only. They do not decide final deletion by themselves. The final delete processors only act on rows where:

SuggestedAction = DELETE
ConfirmDelete = CONFIRM
Safety rule: a file marked DELETE is not deleted until it is also confirmed. The review UI can change the suggested keeper, allow multiple keepers, or clear confirmations before saving the reviewed CSV.

Image duplicate grouping logic

The image finder groups images using a combination of exact file hashing and visual hashing:

  • Exact SHA-256 duplicate: files with the same exact file hash are grouped as exact duplicates.
  • Visual similarity: files can also be grouped when perceptual hashes, aspect ratio, and average color are close enough.
  • Match confidence: groups are labeled Very High, High, or Review Carefully based on visual hash distance to the keeper.

Image suggested keeper preference

For image groups, the initial finder suggestion prefers the file that is most likely to be the best preservation copy:

  1. Higher pixel count / resolution.
  2. Larger file size.
  3. Preferred extension order, such as HEIC/HEIF before JPG/JPEG, then PNG/TIF/TIFF/WebP.
  4. Shorter path.
  5. Alphabetical path tie-breaker.

Every other file in the image group is initially suggested as DELETE, but still requires review and confirmation.

Video duplicate grouping logic

The video finder uses FFprobe and FFmpeg instead of relying only on filenames or file size:

  • FFprobe reads duration, width, height, and file details.
  • FFmpeg samples still frames at configured points in the video, such as 25%, 50%, and 75%.
  • The sampled frames are hashed and compared.
  • Duration difference and sampled-frame similarity drive the duplicate match.
  • Date Modified closeness is only a weak supporting signal, not the deciding factor.

Video suggested keeper preference

The current video preference is designed for compressed/re-exported videos where resolution alone can be misleading.

  1. If all files in the group are in the same directory and have the same non-zero file size, prefer the older Date Modified file.
  2. Otherwise prefer an already archive-renamed video filename.
  3. Prefer a file that already has a matching XMP sidecar.
  4. Prefer higher estimated bitrate / larger effective file size.
  5. Prefer higher resolution.
  6. Use shorter path and alphabetical path as final tie-breakers.
Why bitrate/file size outranks resolution for videos: a very small 1080p video may be heavily compressed and lower quality than a larger 1440x1080 or 720p version. For videos, estimated bitrate and file size are often better preservation signals than resolution alone.

Reviewer-side overrides

Review-SimilarFiles.py may adjust the initial finder suggestions when it opens a CSV. This avoids rerunning expensive video analysis just to change keeper preference.

OverrideWhat it doesWhy
Needs Categorized overrideIf the current keeper is inside Needs Categorized and a duplicate outside that folder is the same size or larger, the reviewer suggests the outside copy as keeper.Helps move away from temporary staging folders.
Video bitrate/file-size overrideFor video groups, the reviewer can prefer the higher estimated bitrate/larger file over a smaller higher-resolution file.A tiny high-resolution video may be over-compressed.
Same folder + same size + older modified overrideIf video files are in the same directory and the file size is the same, the reviewer suggests the older Date Modified file.When quality indicators are equal, the older timestamp may better preserve original file history.
Manual review still matters: these are preservation-oriented suggestions, not guarantees. Always review before confirming deletes.

MPG/MPEG and Video Metadata Direction

MPG/MPEG files are legacy video containers. ExifTool can read useful data from many of them, but it cannot write embedded metadata to MPG/MPEG files.

GoalRecommended Direction
Rename MPG/MPEGYes. Rename from metadata or confirmed CSV date.
Embed Date Taken directly into MPG/MPEGNo. Skip embedded metadata write.
Set Windows filesystem datesOptional helper, using -SetFileSystemDates in rename review processing.
Immich-style date/tags for MPG/MPEGUse the corrective script to create XMP sidecar files beside the videos.
Remux MPG to MP4/MKV just for metadataNot recommended as the default. Sidecars are safer and preserve the original file.

Troubleshooting

PowerShell says ampersand is not allowed

Quote any path containing &, such as "C:\MediaArchiveTools\...".

.\ffmpeg is not recognized

ffmpeg is a folder. Use .\ffmpeg\bin\ffmpeg.exe -version.

Video reviewer shows browser stream disconnect messages

Use the current Review-SimilarFiles.py. Harmless browser-aborted video stream tracebacks are suppressed.

Video preview does not play

The browser may not support that video container/codec. Use Open file from the reviewer card.

Rename script says MPG metadata write is unsupported

That is expected. MPG/MPEG files can be renamed, but embedded metadata writing is skipped. Use filesystem dates and XMP sidecars for those files.

CatchUp found files still needing review

Open photo_rename_catchup_missing_dates.csv, set Decision=CONFIRM, enter DateToUse, then process it with -ReviewCsv.

Need to bulk-confirm duplicate groups

Use Confirm All Remaining Groups. It applies only to currently shown groups, so use the strength/text filters to control scope.

Need video tags/date for Immich

Run Correct-ImageVideoMetadataFromFilename.py. It creates video_file.ext.xmp sidecars for renamed video files using the filename date and parent-folder tag rule.

Final Notes

  • Always use -WhatIf before live rename, delete, or metadata actions.
  • The duplicate review UI never deletes files. It only saves CSV decisions.
  • The duplicate processors delete only confirmed DELETE rows and should use Recycle Bin by default.
  • After renaming, consolidate/combine directories where possible before running de-dupe.
  • Use -CatchUp after organization to rename leftover unrenamed files without ExifTool-scanning the whole completed archive again.
  • Use the central Review-SimilarFiles.py for both image and video review CSVs.
  • The final corrective metadata/tag/sidecar pass should be run after duplicate cleanup.
  • Keep backups before large archive-wide actions.
